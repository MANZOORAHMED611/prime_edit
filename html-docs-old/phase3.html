✓ Feature 18: Macro Recording

Record and playback keyboard sequences for repetitive editing tasks

Implementation Details

  • Component: MacroRecorder singleton
  • Recording: F9 to start, Shift+F9 to stop
  • Playback: F10
  • Storage: Named macros saved in memory

Architecture

class MacroRecorder : public QObject
{
    struct MacroEvent {
        enum Type { KeyPress, KeyRelease, TextInput };
        Type type;
        int key;
        Qt::KeyboardModifiers modifiers;
        QString text;
    };

    bool m_recording;
    bool m_playingBack;
    QVector<MacroEvent> m_currentMacro;
    QMap<QString, QVector<MacroEvent>> m_savedMacros;

public:
    void startRecording();
    void stopRecording();
    void playback(QWidget *target);
    void saveMacro(const QString &name);
    void loadMacro(const QString &name);
};

Recording Process

  1. Press F9 to start recording
  2. Perform editing operations (typing, navigation, etc.)
  3. Press Shift+F9 to stop recording
  4. Press F10 to replay the recorded macro
  5. Optionally save macro with a name for later use

Event Capture

Macros are recorded at the Editor's keyPressEvent() level, capturing all keyboard input including modifiers, special keys, and text input. 

void Editor::keyPressEvent(QKeyEvent *event)
{
    // Record macro event
    MacroRecorder::instance().recordKeyEvent(event);

    // Process event normally
    QPlainTextEdit::keyPressEvent(event);
}

Saved Macros

  • Save current macro with Tools → Save Macro...
  • Load saved macro with Tools → Load Macro...
  • Macros persist in memory during session
  • Future: Save to settings file for persistence

✓ Feature 15: LSP Integration

Language Server Protocol support for code completion, diagnostics, and navigation

Implementation Details

  • Protocol: JSON-RPC 2.0 over stdio
  • Components: LSPClient and LSPManager
  • Languages: Python (pylsp), C++ (clangd), JavaScript/TypeScript (typescript-language-server), Rust (rust-analyzer), Go (gopls)
  • Features: Completion, hover, go-to-definition, find references, diagnostics

Architecture

class LSPClient : public QObject
{
    QProcess *m_process;
    int m_nextId;
    QMap<int, QString> m_pendingRequests;

public:
    void initialize(const QString &rootPath);
    void didOpen(const QString &uri, const QString &languageId, const QString &text);
    void didChange(const QString &uri, const QString &text);
    void completion(const QString &uri, int line, int character);
    void hover(const QString &uri, int line, int character);
    void gotoDefinition(const QString &uri, int line, int character);

signals:
    void completionResult(const QVector<CompletionItem> &items);
    void hoverResult(const QString &content);
    void definitionResult(const Location &location);
    void diagnosticsPublished(const QString &uri, const QVector<Diagnostic> &diagnostics);
};

class LSPManager : public QObject
{
    QMap<QString, LSPClient*> m_clients;

public:
    static LSPManager& instance();
    LSPClient* clientForLanguage(const QString &language);
    void startServer(const QString &language, const QString &rootPath);
};

JSON-RPC Communication

The LSP client implements the Language Server Protocol JSON-RPC 2.0 specification:

  • Message Format: Content-Length header followed by JSON payload
  • Requests: Have unique ID and expect response
  • Notifications: Fire-and-forget messages (didOpen, didChange, etc.)
  • Responses: Matched to requests via ID
